## 创建ReActAgent

~~~java
import io.agentscope.core.ReActAgent;
import io.agentscope.core.message.Msg;
import io.agentscope.core.model.DashScopeChatModel;
import io.agentscope.core.tool.Toolkit;
import io.agentscope.core.tool.Tool;
import io.agentscope.core.tool.ToolParam;

public class QuickStart {
    public static void main(String[] args) {
        // 准备工具
        Toolkit toolkit = new Toolkit();
        toolkit.registerTool(new SimpleTools());

        // 创建智能体
        ReActAgent jarvis = ReActAgent.builder()
            .name("Jarvis")
            .sysPrompt("你是一个名为 Jarvis 的助手")
            .model(DashScopeChatModel.builder()
                .apiKey(System.getenv("DASHSCOPE_API_KEY"))
                .modelName("qwen3-max")
                .build())
            .toolkit(toolkit)
            .build();

        // 发送消息
        Msg msg = Msg.builder()
            .textContent("你好！Jarvis，现在几点了？")
            .build();

        Msg response = jarvis.call(msg).block();
        System.out.println(response.getTextContent());
    }
}

// 工具类
class SimpleTools {
    @Tool(name = "get_time", description = "获取当前时间")
    public String getTime(
            @ToolParam(name = "zone", description = "时区，例如：北京") String zone) {
        return java.time.LocalDateTime.now()
            .format(java.time.format.DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
    }
}
~~~

> 【@ToolParam(name = "zone", description = "时区，例如：北京") String zone】这个有什么作用？
>
> * **如果你不加这个注解**：大模型可能会一头雾水，不知道 `String zone` 到底需要填“北京”、“东八区”、“GMT+8” 还是 “Asia/Shanghai”，这会导致传给你的参数乱七八糟，你的代码就会报错。
> * **加上这个注解后**：相当于给这个参数配了一份“使用说明书”，大模型就会非常精准地提取用户话语中的地点（比如“北京”），并准确无误地塞入 `zone` 这个变量里交给你执行。
>
> **重要**：
>
> * `@ToolParam` 必须显式指定 `name` 属性，因为 Java 运行时不保留方法参数名。





##  ManagerAgent里哪些代码的作用是自动把任务拆成子任务?

定位清楚了。**真正负责"自动拆解子任务"的，是 3 处代码协同**，而且要先说明一个关键点：

> ⚠️ **拆解算法本身不在 `ManagerAgent` 里**，它在 AgentScope 框架内部。`ManagerAgent` 的代码只是「**开启 + 配置 + 触发**」这个能力。

**🔹三处关键代码**

**🔹① 创建计划本子对象**

```java
TripPlan plan = new TripPlan();              // line 61
...
PlanNotebook planNotebook = plan.getPlan();  // line 71  ← 拿到计划本子
```

🔹② 把计划能力挂到 Agent 上 —— **这是开关，最关键的一行**

```java
.planNotebook(planNotebook)
```

**就是这一行**让主管 Agent 具备了"自动拆解任务"的能力。一旦挂上 `PlanNotebook`，框架会自动给这个 ReActAgent 装上一组**内置的"计划工具"**（创建计划 / 拆子任务 / 标记完成 等）。Agent 的大模型在推理时会调用这些工具来完成拆解。

> 注释里也写了（line 88-109）：`PlanNotebook 对象是 Agent 能自主分解任务和步骤执行的核心`。
> 框架还提供了一个更简单的 `.enablePlan()`（用默认配置），本项目用 `.planNotebook(实例)` 是为了**自定义**配置。

**🔹③ 用 sysPrompt 触发拆解行为**

```java
.sysPrompt("""
    你是一个旅游管理主管。
    当用户要求你规划旅游行程时，
    请先创建一个详细的计划，        ← 这句在引导模型去"建计划"
    以及执行计划步骤,
    并对每个计划步骤,要列出擅长执行这个步骤任务的Agent
    然后按计划逐步执行。
    """)
```

光有计划工具还不够，这段提示词**明确要求模型"先建计划、再逐步执行"**，是触发拆解的"指令"。

**🔹配套：拆解的"规则"在 TripPlan 里**

```java
public PlanNotebook getPlan() {
    return PlanNotebook.builder()
        .needUserConfirm(false)   // 拆完不用人工确认，直接执行
        .maxSubtasks(5)           // 最多拆成 5 个子任务  ← 拆解数量上限
        .build();
}
```

**🔹一句话总结分工**

| 谁                                         | 干什么                                   |
| :----------------------------------------- | :--------------------------------------- |
| `TripPlan.getPlan()`                       | 定义**拆解规则**（最多5个、免确认）      |
| `.planNotebook(planNotebook)` （line 117） | **开启**自动拆解能力（核心开关）         |
| `sysPrompt`（line 78-85）                  | **触发**模型去拆解                       |
| **AgentScope 框架内部**                    | 真正**执行**拆解算法（你看不到的那部分） |

所以你在 `ManagerAgent` 里看不到"怎么拆"的具体逻辑——你看到的是**把这个能力打开并配置好**。真正的拆解是运行时由「大模型 + 框架的计划工具」动态完成的，每次拆出来的子任务可能都不一样。

## 但这样不能指定如何拆分的话，它怎么知道要使用routeMaking_agent和tripPlanner_agent呢？

非常好的问题！这正是理解 Agent "智能"的核心。答案是：

> **我们确实"告诉"了它 —— 但不是通过"硬编码的拆分规则"，而是通过「工具描述」。** 模型靠**读工具的描述**，自己判断每个子任务该调谁。

这套机制叫 **工具调用 / 函数调用（Tool Use / Function Calling）**。

**🔹关键代码：`@Tool` 的 description**

看 `RemoteAgentTool.java` 里两个工具方法的注解：

```java
@Tool(description = "擅长制定最优驾车路线")          // line 40public String callRouteMakingAgent(    @ToolParam(name = "prompt", description = "驾车的起点和终点") String prompt) {...}
@Tool(description = "擅长制定旅游景点,饮食,住宿的行程安排")  // line 127public String callTripPlannerAgent(    @ToolParam(name = "prompt", description = "旅游目的地") String prompt) {...}
```

**这两句 `description` 就是模型"知道该用谁"的全部依据。**

**🔹它到底怎么"知道"的？完整链条**

```
① toolkit.registerTool(new RemoteAgentTool())   （ManagerAgent.java:65）
        │ 框架自动扫描 @Tool 方法，
        │ 把【工具名 + 描述 + 参数】登记成一张"能力清单"
        ▼
② Agent 运行时，框架把这张清单塞进大模型的上下文
        │ 模型看到的大致是：
        │   - 工具 callRouteMakingAgent：擅长制定最优驾车路线 (参数:起点和终点)
        │   - 工具 callTripPlannerAgent：擅长制定旅游景点,饮食,住宿的行程安排 (参数:目的地)
        ▼
③ 模型拆出一个子任务，比如"制定北京到天津的驾车路线"
        │ 它做"语义匹配"：这个子任务 ←→ 哪个工具描述最贴近？
        │ "驾车路线" 高度匹配 "擅长制定最优驾车路线"
        ▼
④ 模型决定：调用 callRouteMakingAgent，参数 prompt="北京到天津"
        │ 同理"推荐天津景点美食" → 匹配 callTripPlannerAgent
        ▼
⑤ 工具方法内部，再用 A2A+Nacos 找到真正的远程服务去执行
```

所以"智能"在于：**模型读懂了工具描述，自己做了选择**——不需要你写 `if 路线 then 调A else 调B` 这种规则。

**🔹还有两处"辅助引导"**

**1. sysPrompt 明确提示要"派活给 Agent"**：

```
并对每个计划步骤,要列出擅长执行这个步骤任务的Agent
```

这句话提醒模型：拆完任务后，每一步都要去找一个合适的 Agent（工具）来干。

**2. Agent 自身的 description**（`AgentUtils` 创建时传入）：
比如 RouteMakingAgent 的 `"擅长制定性价比最优的驾车路线"`，会作为它的"智能体卡片(Agent Card)"注册到 Nacos，A2A 发现时也能读到。

**🔹重点：两层不同的"知道"**

很多人会混淆，其实是**两层映射**：

| 层                              | 谁在"知道"           | 靠什么                                                     |
| :------------------------------ | :------------------- | :--------------------------------------------------------- |
| **第1层：选哪个工具**           | 主管的**大模型**     | 读 `@Tool(description=...)` 做语义匹配                     |
| **第2层：工具对应哪个远程服务** | **工具方法内部代码** | `A2aAgent.builder().name("RouteMakingAgent")` + Nacos 发现 |

模型其实**不直接知道** "routeMaking_agent 跑在 8082"——它只知道有个工具叫 `callRouteMakingAgent`、描述是"擅长制定驾车路线"。至于这个工具背后连到哪个服务，是 `RemoteAgentTool.java:50-55` 里的 A2A+Nacos 代码负责的，模型不关心。

**🔹一个实践启示**

既然路由完全靠**工具描述的语义匹配**，那么：

> **描述写得越准、越有区分度，模型选得越对。** 如果两个工具描述含糊或重叠，模型就可能调错。

比如你把描述改成模糊的"处理旅游相关任务"，模型就会分不清该调哪个。这也是为什么 prompt/描述工程在 Agent 开发里很重要。

------

简单说：**拆分是框架自动的，但"每块派给谁"是模型读着你写的工具描述决定的**——你通过描述间接"指定"了规则，只是用自然语言而非 if-else。





**⭐要是不写这个sysPrompt的话，它就不会去自动分配任务了吗？**

（你说的"不屑"应该是"不写" 😄）

答案是：**不会"完全不分配"，但会变得很不可控。** 关键在于要区分「**能力**」和「**引导**」是两码事。

🔹先理清：谁负责什么

| 职责                       | 由谁决定                      | 删了 sysPrompt 还在吗 |
| :------------------------- | :---------------------------- | :-------------------- |
| **能不能拆任务**           | `.planNotebook()` (line 117)  | ✅ 还在                |
| **选哪个工具/Agent**       | `@Tool(description)` 工具描述 | ✅ 还在                |
| **要不要先规划、每步派活** | **sysPrompt**                 | ❌ 没了                |

**sysPrompt 不是"开启分配能力"的开关**——分配能力来自工具 + 计划本子。sysPrompt 是"**行为引导**"：告诉模型该以什么角色、按什么套路去工作。

🔹所以删掉 sysPrompt 会怎样？

**不会彻底罢工**，因为：

- ReActAgent 的本能就是"为完成任务去调用手头的工具"。你问"规划北京到天津路线"，它看到有个工具描述是"擅长制定驾车路线"，**多半还是会去调** `callRouteMakingAgent`——这跟 sysPrompt 无关。
- `.planNotebook()` 还挂着，框架自带的计划工具和内置提示也还在。

**但行为会变得不稳定、不可控**，可能出现：

1. **不先做计划就乱答**：少了"请先创建一个详细的计划"，模型可能跳过规划，直接给个粗略答案。
2. **不派活、自己硬答**：少了"列出擅长执行的Agent…按计划逐步执行"，模型可能**用自己的知识**直接编一个行程，而**不去调用**两个专家 Agent。
3. **丢失角色设定**：少了"你是一个旅游管理主管"，它就不知道自己是个"调度者"，可能表现得像个普通问答助手。
4. **每次行为飘忽**：同样的问题，这次调了专家、下次没调，一致性差。

🔹一个比喻

- **`.planNotebook()` + 工具** = 给员工配齐了"会议室"和"各部门通讯录"（**能力**）。
- **sysPrompt** = 给员工的"岗位说明书"：你是主管，遇到项目先拆解、再分派给对应部门、最后汇总（**规矩**）。

撤掉岗位说明书，员工**还能**打电话给各部门（能力还在），但他可能**不按流程**——也许自己闷头干了，也许随便派，全凭"本能"。

🔹结论

> 删 sysPrompt ≠ 关闭任务分配。任务分配靠的是**工具描述 + ReAct 本能**。
> 但 sysPrompt 决定了它**是否稳定地"先规划、再逐步委派、最后汇总"**。少了它，多智能体协作的可靠性会明显下降，甚至退化成"一个 Agent 自问自答"。





## AgentScope框架将智能体卡片以Bean的形式自动注入到注册中心

SpringBoot如何知道我要注入的注册中心是nacos，而不是其他的注册中心呢？

* 答案就是在配置文件里，配置文件的这里就指明了nacos的地址，因此SpringBoot就能够自动的将这个Bean写入到这个nacos里

  ~~~yaml
  # A2A协议服务端  
        server-addr: localhost:8848
  ~~~

* 需要注意的是：要写入到注册中心里的这个Bean，返回对象不能是void，返回的应该是智能体，也就是ReActAgent对象。

* 以Bean方式注入的缺点就是：不能像手动写入注册中心一样，可以自定义卡片。

* 同时能自动将ReActAgent的这个Bean自动注入到nacos里，是因为这个依赖

  * 但主管Agent是不需要这个依赖的，因为它不需要注入到nacos里，因为导入了这个依赖就要提供这个Bean，主管Agent只需要从nacos里读取即可。

  ~~~xml
    <!--   以 SpringBoot 方式添加 A2A 依赖     -->

              <groupId>io.agentscope</groupId>
              <artifactId>agentscope-a2a-spring-boot-starter</artifactId>
  ~~~

## 在java里，不是@Configration注解的类才能自定义bean吗，为什么@Component注解的class也可以自定义bean？

不是"只有 `@Configuration` 类才能定义 Bean"。准确的说法是：

> 只要一个类**本身被 Spring 管理**（被 `@Component`、`@Configuration`、`@Service`、`@Import` 等纳入容器），它里面的 `@Bean` 方法就会被解析、注册成 Bean。 

`@Configuration` 底层也是个 `@Component`（点进去看注解就知道）。所以两者都能定义 Bean。**真正的区别不在"能不能"，而在"语义不同"** —— 这就是 Spring 文档里说的 **Full 模式 vs Lite 模式**。
